ハッシュ法

ハッシュ法（hashing）は，データの量にかかわらず挿入，探索，削除の操作を実質的にO(1)O(1)で行える優れたアルゴリズムです． キーの値をデータが格納される位置（配列の添え字の値）に関連づけるのがハッシュ法です．

ハッシュ法の原理

ある特定の範囲内に収まる整数をキーとする一連のデータがあるとします． また，話を簡単にするため，キーの重複はないものとします． このデータを表に登録して，探索を行うことを考えます． 大きさ100の配列tableを用意して，キーの値xをこの配列の添え字とします． つまり，キーが0のデータはtable[0]，キーが37のデータはtable[37]とします． この方法では，挿入，探索，削除ともに，O(1)O(1)で行えることは明らかです．

ところで，一般的にはキーを配列の添え字にできるケースはそう多くはありません． 例えば，本学の学籍番号を考えてみましょう． 本学の学生番号はxxFIyyyとなっており，xxには西暦の下2桁，yyyは001から始まる番号が入ります． これを学科全学年のリストを作ろうとすると，下3桁の番号が連続ではなく，特に700番台以降は使われません． また，学部全体のリストを作ろうとすると，FI，FA，FRでこれも連続ではありません．

そこで，キーの値xxを配列の添え字へと写像する関数h(x)h(x)を導入します． この関数をハッシュ関数（hash function）と呼びます． また，ハッシュ関数が返す値をハッシュ値（hash value）といいます． ハッシュ（hash） とは，「（肉などを）細かく切り刻む」という意味で，キーの値を「切り刻む」ことによって，配列の添え字に対応づけます．

ハッシュ関数h(x)h(x)があれば，キーの値をもとにして，配列にデータを絡納することができます． データを格納する配列tableがあったとすると，キーxxをもつデータは，配列の要素table[h(x)］に格納されます． データを格納する配列をハッシュ表（hash table）と呼び，ハッシュ表の各要素をバケット（bucket）と呼びます． キーの値からハッシュ関数を用いてハッシュ値を求めて，ある特定のバケットに結びつけることによって，高速な探索を実現するのがハッシュ法です． 計算量がO(1)O(1)のハッシュ関数を用いれば，挿入，探索，削除をO(1)O(1)で行うことができます．

ハッシュ値の衝突

ここで，文字列をキーとするデータを，大きさが100の配列に格納することを考えましょう． ハッシュ関数は文字列中のすべての文字コードを加算するものとすると，以下のようになります． このハッシュ関数は100の剰余をハッシュ値としているので，ハッシュ値は必ず0～99となります． 

int hash(String s) {
  int sum = 0;
  
  for (int i = 0; i < s.length; i++) {
    sum += (int)s.charAt(i);
  }
  return sum % 100;
}

実際には衝突が発生したとしても，それを回避して正しく探索を行う方法が知られています． しかし，頻繁に衝突が発生すると，探索に時間がかかり高速性というハッシュ法の利点が失われてしまいます． つまり，すべてのバケットにできるだけ均等に振り分けるようなハッシュ関数を選ばなければなりません．

衝突の対策には2通りの方法があります． 一つ目の方法は，同じハッシュ値のデータを連結リストで繋ぐというものです． これをチェイン法（chaining）と呼びます． もう一つの方法は，ハッシュ値に対応するバケットが既にふさがっていたら，ある手順に従って別のバケットにデータを格納するというものです． これをオープンアドレス法（open addressing）と呼びます．

衝突対策1：チェイン法

チェイン法は，同じハッシュ値のデータを連結リストで繋ぐという方法です． Fig.8.1のように，ハッシュ表には連結リストへのリンクを入れ，データはリンクで繋いでおきます． Fig.8.1では，バケット0（ハッシュ値が0）にはA，Bという2つのデータが登録されています． また，バケット1は空です． 探索を行うには，まずハッシュ値を計算しバケットを決めてから，そのバケットに繋がれている連結リストを線形探索します． 線形探索には連結リストの長さに比例した計算量がかかるので，連結リストは短い方が良いです． 挿入と削除については，ハッシュ値を求めてバケットを決めてから，連結リストへの挿入と削除で行います．

ここでチェイン法の性質について考えてみましょう． バケットの数をBB，登録するデータの数をNNとします． また，ハッシュ関数はデータをすべてのバケットに均等に振り分けるようなハッシュ値を生成するとします． この時，各バケットには平均N/BN/B個のデータが入ることになります．

チェイン法の計算量

探索を行うには，まずハッシュ値を求めて，どのバケットを探すべきかを決めます． この計算量はO(1)O(1)です． 次に，連結リストの線形探索は，リスト長の平均はN/BN/Bなので，計算量はO(N/B)O(N/B)になります． したがって，探索全体の計算量はO(1+N/B)O(1+N/B)となります．

挿入の計算量は，連結リストに要素を挿入するだけなのでO(1)O(1)です． しかし，キーの重複チェックのために探索を行わなければならないので，計算量は結局O(1+N/B)O(1+N/B)になります． 削除についても，連結リストからデータを削除すること自体はO(1)O(1)ですが，対象となるデータを探し出さなくてはならないので，結局，計算量はO(1+N/B)O(1+N/B)になります．

ところで，O(1+N/B)O(1+N/B)という計算量は，実際にどの程度なのかを考えてみましょう． データの個数NNに対してバケットの数BBが十分に大きければ，実質的にはN/BN/Bを定数と見なすことができるので，O(1+N/B)=O(1)O(1+N/B)=O(1)となります． 一方で，データの個数NNに対してバケットの数BBが小さい場合，N/BN/Bは支配的になります． つまり，O(1+N/B)=O(N/B)=O(N)O(1+N/B)=O(N/B)=O(N)となります． このような計算量になると，各操作の実行時間はデータの個数に比例します．

衝突対策2：オープンアドレス法

オープンアドレス法は，衝突が発生したときに，ある手順に従って別のバケットにデータを格納します． この手順のことを再ハッシュ（rehashing）といいます． チェイン法では，データそのものではなく，連結リストへのリンクをハッシユ表に登録しました． これに対して，オープンアドレス法では，ハッシュ表のバケットにはデータそのものが登録されます．

キ－xxのハッシュ値h(x)h(x)で示されるバケットが既にふさがっている場合，再ハッシュ手順に従って別のバケットを調べます． ここで，再ハッシュをkk回行って得られるハッシュ値をhk(x)hk(x)とします． 再ハッシュ手順は，バケットh1(x)h1(x)，h2(x)h2(x)，h3(x)h3(x)，⋯⋯を順番に調べることです． 再ハッシュ手順の途中で，空のバケットがあった場合，そのバケットにデータを登録します． もし，再ハッシュをBB回（BBはバケットの個数）繰り返しても，空のバケットが見つからなければ，ハッシュ表が満杯なので，これ以上データを登録することができなくなります．

探索は，まずキ－xxのハッシュ値h(x)h(x)のバケットを調べ，もしそのバケットのデータがキ－xxならば，それが探しているデータになります． そうでなければ，再ハッシュ手順に従って，バケットh1(x)h1(x)，h2(x)h2(x)，h3(x)h3(x)，⋯⋯を順番に調べ，キ－xxを持つデータがあれば，それが探しているデータになります． もし途中で空のバケットに出合ったら，キ－xxを持つデータは存在しません．

再ハッシュ手順として，いくつかの方法があります． ハッシュ値h(x)=h0(x)h(x)=h0(x)として，k≥1k≥1とすれば，以下の方法があります．

  1. hk(x)=(hk−1(x)+1)modBhk(x)=(hk−1(x)+1)modB
  2. hk(x)=(hk−1(x)+ck)modBhk(x)=(hk−1(x)+ck)modB（ccは定数）
  3. hk(x)=(hk−1(x)+nk)modBhk(x)=(hk−1(x)+nk)modB（nknkは乱数列）
1の方法は最も簡単ですが，衝突が起きると連続するバケット（塊）がふさがってしまうため，さらに衝突が起きやすくなります． 2の方法は1の方法よりもバケットの塊ができにくいですが，cc個おきにバケットがふさがってしまうので，依然として衝突は起きやすくなってしまいます． 3の方法は，あらかじめ1～B−1B−1までの数が必ず1回だけ出現する乱数列を用意しておく方法で，ハッシュ値が異なれば，再ハッシュされたハッシュ値が異なるように工夫されているため，塊が起きにくくなっています． 以下の説明では，簡単のために，再ハッシュ手順として1の方法を用いることとします． なお，オープンアドレス法では，再ハッシュとの兼ね合いから，バケットの個数BB（ハッシュ表の大きさ）は素数，あるいは2n−12n−1（nnは整数）とするのが望ましいとされています．

オープンアドレス法の実例

次に，オープンアドレス法の実例を考えてみましょう． Fig.8.2(1)に示すデータのうち，a～eの5つのデータをこの順序で登録します．

まず，最初にデータa（キーの値がaであるデータ）を登録します． aのハッシュ値は1であり，バケット1は空なので，データaをバケット1に登録します． 同様にして，データb，cをそれぞれバケット5と8に登録します． 次にデータdを登録しますが，dのハッシュ値は5で，データbと衝突します． そこで，先ほどの再ハッシュ手順から，h1(d)=(h(d)+1)mod10=(5+1)mod10=6h1(d)=(h(d)+1)mod10=(5+1)mod10=6となります． バケット6は空なので，データdをバケット6に登録します． 最後にデータeを登録しますが，バケット6は既にふさがっているので，再ハッシュを行います． h1(e)=(h(e)+1)mod10=(6+1)mod10=7h1(e)=(h(e)+1)mod10=(6+1)mod10=7となり，バケット7は空なので，データeをバケット7に登録します． この時点でハッシュ表の内容はFig.8.2(2)のようになります．

この状態でデータbを探索してみましょう． bのハッシュ値は5なので，まずバケット5を調べます． バケット5にはデータbが登録されているので，探索は成功です．

次に，データeを探索してみましょう． eのハッシュ値は6ですが，バケット6に登録されているデータはdですので，探索対象のキーではありません． そこで，再ハッシュ手順に従って，バケット7，8，・・・を順番に調べていくと，バケット7にeが入っているので，探索は成功となります．

では，データfを探索するとどうなるでしょうか． fのハッシュ値は6ですが，バケット6に登録されているデータはdですので，再ハッシュ手順を行います． 再ハッシュ手順に従って，バケット7，8，・・・を順番に調べると，バケット7にはe，バケット8にはcが入っており，バケット9は空です． ここで，バケット9は空なので，データfはハッシュ表には登録されていないことがわかり，探索は終了します．

次に，削除について考えてみましょう． Fig.8.2(2)の状態からデータdを削除してみましょう． 単純にデータdの入っているバケットを空にすると，Fig.8.2(3)の状態になりますが，この状態では探索がうまくいかなくなります． 例えば，Fig.8.2(3)のハッシュ表に対して，データeを探索することを考えます． eのハッシュ値は6なのでバケット6を調べますが，空になっているので，データeは見つからずに，探索は終了となります． しかし，先に見たように，実際にはデータeはバケット7に入っています．

このような問題が起きるので，削除をする場合には，データを空にするのではなく，「削除済み」という状態を設定します． そして，探索を行うときに，「削除済み」のバケットがあれば，探索を継続することにします． このようにしてデータdを削除すると，Fig.8.2(4)のようになります． Fig.8.2(4)の状態でデータeの探索を考えましょう． バケット6を調べるときに，今度は「空」ではなく「削除済み」になっているので，さらに探索を継続し，バケット7でデータeを見つけ出すことができます．

挿入を行うときには，「削除消み」のバケットは，「空」のバケットと同様に扱います． 例えば，Fig.8.2(4)の状態でデータfを挿入するとき，fのハッシユ値は6なので，バケット6を調べます． バケット6には「削除済み」のマークがついているので，ここにデータfを登録します．

ハッシュ関数

ハッシュ法では，均等なハッシュ値を発生するハッシュ関数が求められます． ここでいう「均等」とは，全てのキー値を対象としたときに，ハッシュ値が等確率で生成される（静的均等）だけでは不十分です． 実際の使用にあたっては，全てのキー値が等確率で出現するわけではなく，キー値はかなり偏っているのが一般的です． そのため，実際にキーとして使われる値に対して，ハッシュ値が均等に振り分けられる（動的均等）ことが望まれます．

ハッシュ関数は，できるだけキーの全てのビットがハッシュ値に影響を与えるようにします． 「キーの値を二乗してその中央をとる（平方採中法，middle-suqare method）」という方法は，かなり良好な結果が得られることが知られています．
