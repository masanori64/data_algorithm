クイックソート，マージソート，ビンソート

クイックソート

クイックソート（quick sort）は，C.A.R. Hoareが26歳の時（1960年）に開発したアルゴリズムです． クイックソートの計算量はO(nlogn)O(nlog⁡n)であり，ソートアルゴリズムの中でも最速の部類です．

クイックソートのアルゴリズムは次の通りです． n個の要素a[0], …, a[n-1]を持つ配列aをソートすることを考えましょう． まず，ある適当な要素を1個選び出し，これを要素xとします． それを最終的に置かれるべき位置a[v]に移動します． この要素xのことをピボット（pivot，枢軸）と呼びます． この時，xより大きな要素はa[v]より右に，xより小さな要素はa[v]より左へと振り分けます． これにより，Fig.14.1に示すように，配列aは次の3つの部分に分けられます．

  1. a[v]よりも小さい要素 a[0], a[1], …, a[v-1]
  2. a[v]
  3. a[v]よりも大きな要素 a[v+1], a[v+2], …, a[n-1]

a[v]よりも左側にはa[v]よりも小さな要素，右側にはa[v]よりも大きな要素なので，それぞれの部分についてソートを行えば，配列a全体のソートが完了します． この時，1や3の部分についても，いま説明した方法で3分割することを再帰的に行います． このような，大きな問題を複数の小さな問題に分割して，それぞれを解いていく方法は，分割統治法（divide-and-conquer method）と呼ばれ，アルゴリズムのテクニックの1つです．

クイックソートのアルゴリズムは以下の通りになります． ここで，quicksortメソッドは，配列aのうちa[l]～a[r]の要素をソートします． 

void quicksort(int[] a, int l, int r) {
  if (l>=r) return;                             // (1)
  
  適当な要素a[v]をピボットとする                // (2)
    a[v]よりも小さい要素をa[l]～a[v-1]に集める
    a[v]よりも大きな要素をa[v+1]～a[r]に集める
  
  quicksort(a, l, v-1); // 左の部分をソート        (3)
  quicksort(a, v+1, r); // 右の部分をソート        (4)
}

このメソッドの中で(2)の部分は複雑です． この部分を特に分割（partitioning）と呼び，クイックソートの処理のほとんどを占めます． この分割アルゴリズムは巧妙で高速です． クイックソートが高速なのは，この分割の効率の良さによるものです．

分割アルゴリズム

Fig.14.2(1)の配列を例として，分割のアルゴリズムを考えてみましょう． 分割アルゴリズムでは，2つのポインタを使って，配列の両端から中央に向かって走査します． 左端から右に向かって進むポインタをi，右端から左に向かって進むポインタをjとします．

まず，分割アルゴリズムが，ピボットになるデータが配列の端にあることを前提としているので，配列の右端の要素30をピボットとします． つまり，配列全体を「30より大きい要素」と「30より小さい要素」に振り分けます． 次に，ポインタiとjを初期化します． ポインタiは左端の要素55，jはピボットの隣の要素46を指すようにします． ポインタiとjをそれぞれ内側に向かって進めていき，ポインタiの左側には30よりも小さい要素のみが，ポインタjの右側には30よりも大きい要素のみが存在するようにします． 具体的な手順は以下の通りです．

  1. ピボット30よりも大きな要素が見つかるまで，ポインタiを右ヘ進める
  2. ピボット30よりも小さな要素が見つかるまで，ポインタjを左ヘ進める
  3. ポインタiが指す要素とポインタjが指す要素を入れ替える
  4. ポインタiとjがぶつかるまで1～3を繰り返す
  5. 最後に，ポインタiが指している要素をピボット（右端の要素）と交換する

このように，2つのポインタiとjを使って要素を「はさみうち」にすることで，配列の要素を「ピボットよりも小さい部分」と「ピボットよりも大きい部分」に分割することができます．

クイックソートの計算量

クイックソートの平均的な計算量はO(nlogn)O(nlog⁡n)です． また，クイックソートは計算量がO(nlogn)O(nlog⁡n)のアルゴリズムの中で，最も高速なアルゴリズムです． この理由は，ポインタiとjを動かす際に比較する対象が常にピボットで固定されているため，配列の要素同士を比較するよりも高速に比較ができます．

クイックソートが最も高速に実行されるのは，分割を行った結果，ピボットの要素がちょうど真ん中に位置するような場合です． 反対に，ピボットが端に来るような分割が続くと，クイックソートはかなり効率が悪くなります． 皮肉なことに，あらかじめ整列されている配列をソートさせると，クイックソートの効率は最悪となり，計算量はO(n2)O(n2)になります．

マージソート

マージソート（merge sort）は，計算量O(nlogn)O(nlog⁡n)の高速なソートアルゴリズムですが，定数係数が大きいため，クイックソートには太刀打ちできません． しかしながら，マージソートは他のソートアルゴリズムとは違い，要素をシーケンシャルアクセスしてソートするという特徴があります．

マージ処理

まず始めに，マージソートの基礎となるマージ（merge）処理について説明します． ソート済みの2個のデータ列を，ソート済みの1個のデータ列にまとめる操作をマージといいます．

2つのソート済みのデータ列aとデータ列bをマージして，データ列cにまとめることを考えます． まず，列aと列bの先頭の要素を比較して，そのうち小さい方を列から取り除いて列cに追加します． これを，列aか列bのどちらか一方の要素がなくなるまで繰り返します． 最後に，残された要素を列cの末尾に順に追加することで，マージは完了します．

具体的なマージの手順を考えてみましょう． ここでは，それぞれ3個の要素を持つソート済みのデータ列aとデータ列bをマージします． 列aは30, 46, 95，列bは15, 26, 90となっており，それぞれ要素は昇順にソートされています． Fig.15.1(1)がマージを行う前の状態です．

まず，列aの先頭の要素30と列bの先頭の要素15を比較します． 15の方が小さいので，列bから15を取り出して列cに入れると，Fig.15.1(2)になります． 再び列aと列bの先頭同士を比較すると，今度も列bの先頭の要素26の方が小さいので，列bから26を取り出して列cに入れると，Fig.15.1(3)になります．

同様にして，列aから要素30と46を取り出して，列cに追加すると，Fig.15.1(4)～(5)になります． そして，列bから90を取り出して列cに追加します． Fig.15.1(6)の時点で，列bは空になります． そこで最後に，列aに残された要素95を列cの末尾に追加すれば，Fig.15.1(7)となり，マージ処理は完了です．

マージソートのアルゴリズム

マージソートの手順は以下の通りです．

  1. データ列を真ん中で2つの部分列a, bに分割する
  2. 部分列a, bをそれぞれソートする
  3. ソート済になった部分列a, bをマージする

手順2でのソートには，マージソートを再帰呼び出しして利用します． 再帰呼び出しは，部分列の長さが1になった時点で終了します． このような「データを分割して個別に処理する」という手法は，典型的な分割統治法です． 以下に，マージソートのアルゴリズムを示します． 

void mergesort(int[] a) {
  if (a.length == 1) return;                 // (1)
  
  列aを2つの列x, yに分割する                 // (2)
  mergesortを再帰呼び出しで，列xをソートする // (3)
  mergesortを再帰呼び出しで，列yをソートする // (4)
  列x, yをマージして，列aにする              // (5)
}

マージソートの具体例として，Fig.15.2に示す8個の要素をマージソートすることを考えてみましょう． まず始めに，長さ8の列に対してマージソートを行います． 長さ8の列は，長さ4の部分列2個に分割されて，それぞれ再帰的にマージソートを行います． そして，長さ4が長さ2へ，さらに長さ2が長さ1へと分割されて，再帰的にマージソートが行われていきます．

長さ1まで分割されたら，今度はマージを行います． まず，長さ1の部分列2個をマージして，長さ2の部分列にします． 次に，長さ2の部分列2個をマージして，長さ4の部分列にします． 最後に，長さ4の部分列2個をマージして，長さ8の列にした時点で，マージソートは完了です．

Fig.15.2の(1)～(4)が分割で，(4)～(7)がマージになります． ここで，分割においては列を分割するだけで，要素の並び替えを行っていません． 対して，マージでは要素の位置が入れ替えられて，要素がソートされていきます．

マージソートの計算量

マージソートの分割では，部分列の長さが半分になっていきます． 分割を行うごとに部分列の長さが半分になるので，長さnn個のデータに対して，log2nlog2⁡n回の分割を行うことになります． 分割の計算量は，どのようなデータ構造を用いるかで異なります． 例えば，配列を用いる場合，上限と下限を足して2で割るだけで良いため，分割1回あたりの計算量はO(1)O(1)になります． 対して，連結リストを用いる場合，連結リストの要素の個数を数えながら，真ん中を探す必要があるので，分割1回あたりの計算量はO(n)O(n)になります． 結果として，分割の計算量は，配列の場合はO(logn)O(log⁡n)，連結リストの場合はO(nlogn)O(nlog⁡n)となります．

マージは分割と同様に，log2nlog2⁡n回実行されます． マージでは，マージ対象の要素の個数分だけ処理が行われますので，マージ1回あたりの計算量はO(n)O(n)になります． つまり，マージ全体の計算量は，O(nlogn)O(nlog⁡n)となります．

マージソート全体の計算量は，分割の計算量とマージの計算量の大きな方を採用しますので，結果としては，配列でも連結リストでもO(nlogn)O(nlog⁡n)の計算量となります．

ところで，計算量O(nlogn)O(nlog⁡n)はソートアルゴリズムの下限ですので，マージソートはクイックソート同様に非常に高速なソートアルゴリズムです． しかし，マージソートは定数係数が大きいため，クイックソートには及びません．

ところで，クイックソートは最悪の場合，計算量がO(n2)O(n2)になります． これは分割する際のピボットの値によって，分割統治が有効に働かなくなるためです． 対照的に，マージソートは要素の値には関係なく，列の個数が等しくなるように分割を行うので，クイックソートのようにソート前のデータの並びに影響を受けません． つまり，マージソートは常にO(nlogn)O(nlog⁡n)の計算量でソートできることが保証されています．

ビンソート（バケットソート）

比較によらないソートアルゴリズム

これまで紹介したアルゴリズムは，いずれもキーの大小を比較して，その結果に基づいてレコードをソートするものでした． キーの大小比較を利用する限り，ソートアルゴリズムの計算量の下限はO(nlogn)O(nlog⁡n)になることが理論上証明されています．

一方，キーの大小比較によらない整列アルゴリズムがいくつか存在します（例えば，ビンソート（bin sort），分布数え上げソート（distribution counting sort），基数ソート（radix sort））． これらの計算量はいずれもO(n)O(n)であり，O(nlogn)O(nlog⁡n)の壁を超えることができます．ただし，その代償として，大きな作業領域（いずれもO(n)O(n)の領域計算量）を必要とします． また，これらのアルゴリズムを利用するには，キーがある特定の条件を満たす必要があります． さらに，計算量のオーダーが低いとはいえ，実用的には，O(n)O(n)とO(nlogn)O(nlog⁡n)の違いは，さほど大きなものでもありませんので，実際のデータを使って実験したうえで，アルゴリズムを選択することになります．

ビンソートのアルゴリズム

さて，ここでは，比較によらないソートアルゴリズムの代表例として，ビンソート（bine sort）を紹介します． ある範囲内に収まっている整数（例えば，0～100の整数）を整列することを考えます．整列の対象となる整数は重複していないものとします．

ビンソートの手順は以下の通りです．

  1. データが取りうる値の種類の数と同数のビンを用意し，それぞれのビンにデータの値に対応する名前を付けておく
  2. 整列対象となるデータを1つずつ順に，対応する名前のビンに入れていく
  3. すべてのデータをビンに入れ終えた時点で，ビンの中身を順番に取り出して並べる
つまり，ビンを利用して，データの値によって仕分けするわけです．

ビンソートの具体例をFig.17.1に示します．整列データの範囲は，1から10までとします．また，整列するデータは7, 4, 2, 8 ,1とします． そして，以下の手順で，ソートを実行します．

  1. Fig.17.1 (1)：1～10個のビンを用意します．それぞれのビンには名前を付けておきます．
  2. Fig. 17.1 (2)：整列するデータ7, 4, 2, 8, 1を，それぞれ対応する名前が付いたビンに入れます．
  3. Fig. 17.1 (3)：小さい値の名前が付いたビンから順に中身を取り出します．

ビンソートの計算量

データの個数をnn，用意したビンの個数をmmとして，ビンソートの計算量を考えてみましょう． nn個のデータをビンに振り分けるにはO(n)O(n)の手間がかかります．また，データをビンから（昇順に）取り出すには，すべてのビンを順にチェックしないといけないので，O(m)O(m)の手間がかかります．したがって，ビンソートの計算量はこれらを合計したO(m+n)O(m+n)になります．用意したビンの個数mmがデータの個数nnに比べて極端に大きい場合を除けば，ビンソートに必要な計算量はO(n)O(n)とみなすことができます．

キーの範囲が広すぎる場合（例えば，n=10,m=100000n=10,m=100000）には，ビンソートは実用的ではありません． また，作業用にmm個のビンを使うということは，言い換えればO(m)O(m)の作業領域が必要となります．時間計算量の時と同様に，mmがnnに比べて極端に大きいケースを除けば，要素の個数nnに比例する作業領域が必要ということになり，ビンソートの領域計算量はO(n)O(n)とみなせます．

ビンソートの性質

ビンソートは計算量がO(n)O(n)と高速である反面，データの量に比例する作業領域が必要となります．時間と空間のトレードオフということができます．キーの大小比較を利用するアルゴリズムの計算量の限界O(nlogn)O(nlog⁡n)を超える代償として，大きな作業領域が必要になるわけです．

ビンソートには，次の2つの弱点があります．

  1. キーの重複が許されない．：
    連結リストなどを使用し，ビンに複数のデータが入るようにすれば解決可能．なお，分布数え上げソート（詳細は教科書を参照のこと）は，連結リストを使用せずに重複したキーをソート可能．
  2. キーはある範囲に収まる整数でなければならない（実用上，その範囲はかなり狭い必要有）：
    分布数え上げソートも同様の制約を受ける．基数ソートアルゴリズム（詳細は教科書を参照のこと）では，この制約を逃れることが可能．
